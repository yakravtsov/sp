# Пошаговый план разработки прототипа приложения (1 разработчик)

Цель: рабочий прототип управления камерой Yi Action Camera по USB-отладке на Android-устройстве: подключение к камере, управление настройками, скачивание файлов.

> Ограничения: без тестов, без enterprise-процессов, без релизных пайплайнов. Фокус — быстро собрать рабочий end-to-end прототип.

## 0) Рамки и формат прототипа
1. Выбрать стек: Android-приложение (Kotlin), минимум зависимостей.
2. Определить минимальный UX:
   - Экран подключения.
   - Экран live/состояния камеры (с отображением RTSP-потока).
   - Экран настроек.
   - Экран файлов (листинг + скачивание).
3. Сразу договориться о поддерживаемой модели: J22 (`YDXJ_v22`) и протоколе TCP JSON на `192.168.42.1:7878`.

## 1) Каркас Android-проекта
1. Создать проект с 4 слоями:
   - `app/ui`
   - `domain`
   - `data/camera`
   - `data/network`
2. Добавить необходимые permissions (Wi‑Fi/network/storage в рамках актуального Android API).
3. Настроить запуск на телефоне в режиме USB debug (ADB):
   - `adb devices`
   - `adb install -r ...`
4. Добавить экран «Технический статус» (лог событий подключения и команд), чтобы быстрее отлаживать протокол.

## 2) Сетевой базис под Wi‑Fi камеры
1. Реализовать Network Binding для Android 23+:
   - `NetworkRequest` с `TRANSPORT_WIFI`.
   - `removeCapability(NET_CAPABILITY_VALIDATED)`.
   - bind/unbind процесса к сети камеры.
2. Реализовать проверку, что активна сеть камеры (SSID камеры / доступность `192.168.42.1`).
3. Сделать контроллер подключения с ретраями:
   - Wi‑Fi retry до 3 попыток.
   - таймаут подключения 7000 ms.
4. Вывести состояние в UI: `Disconnected / Connecting / Connected / Session active / Error`.

## 3) Транспортный слой TCP (ядро прототипа)
1. Реализовать `CameraSocketClient` для `192.168.42.1:7878` с параметрами:
   - connect/read timeout ~2000 ms,
   - keepAlive, tcpNoDelay,
   - reconnect до 3 попыток.
2. Реализовать однопоточную модель обработки I/O:
   - одна очередь команд FIFO,
   - одновременно только 1 команда «в полете».
3. Добавить буферизацию входящих данных (`StringBuilder`) и разбор нескольких JSON из одного пакета.
4. Сделать dispatcher входящих сообщений по `msg_id`:
   - `257`, `258`, `1793`, `7`, и ответы на текущую команду.
5. Реализовать heartbeat в двух направлениях:
   - ответ на входящий `1793`.
   - исходящий `16777244` каждые 5 секунд (для J22), с перезапуском таймера после отправки команд.

## 4) Сессия и инициализация камеры
1. Ввести `SessionManager`:
   - `Start Session` (`257`, token=0).
   - сохранение token из `param`.
2. Проверить флаги состояния из ответа `257`:
   - `album`, `fwupdate`, `mvrecover`, `sdformat`, `sdoptimize`, `usbstorage`, `voice_control`, `live`.
3. После успешного старта выполнить init flow:
   - `msg_id=3` (Get All Settings),
   - синхронизация `camera_clock` через `msg_id=2`,
   - для J22 выполнить `Reset VF` (`259`, `param=none_force`),
   - подгрузить options (`msg_id=9`) для ключевых настроек.
4. На выходе иметь состояние `CameraReady` для UI.

## 5) Экран и сценарий «Подключение + live состояние»
1. Кнопка `Connect` запускает цепочку:
   - bind network → TCP connect → start session → init.
2. Показать ключевые данные:
   - модель,
   - battery/adapter (через `msg_id=13` и/или нотификации),
   - SD card status,
   - app status / record status.
3. Обработать нотификации `msg_id=7` и сразу отображать в логе/статусе UI.
4. Добавить `Disconnect` с корректным `Stop Session` (`258`) и очисткой состояния.

## 5.1) Отображение live-потока с камеры (обязательно для прототипа)
1. После успешного init запускать live-preview сценарий:
   - отправить `Reset VF` (`259`, `param=none_force`) для J22,
   - открыть RTSP: `rtsp://192.168.42.1/live`.
2. Добавить на экран live-плеер с базовыми контролами:
   - `Start Live`, `Stop Live`,
   - индикатор состояния потока (`Buffering / Playing / Error`).
3. При обрыве сети/сессии автоматически останавливать плеер и показывать причину.
4. При переподключении автоматически пытаться поднять live снова (после восстановления сессии).
5. Для отладки вывести отдельные логи live-контура:
   - запуск/остановка RTSP,
   - ошибки декодера/соединения,
   - длительность восстановления после reconnect.

## 6) Управление настройками камеры
1. Создать реестр поддерживаемых для прототипа настроек (не все сразу):
   - видео: `video_resolution`, `video_quality`, `video_standard`, `warp_enable`;
   - фото: `photo_size`, `photo_quality`;
   - системные: `camera_clock` (служебно).
2. Для каждого параметра:
   - получить options (`msg_id=9`),
   - показать в UI selector,
   - менять через `msg_id=2`.
3. После изменения с `related > 0` обновлять settings (`msg_id=3`) и affected options.
4. Для `warp_enable` выполнять `Reset VF` (`259`) после изменения.

## 7) Файлы: листинг и скачивание
1. Реализовать API листинга:
   - `msg_id=1282` для `/tmp/fuse_d/DCIM`, пагинация `from/to`.
2. Отрисовать список файлов в UI:
   - имя,
   - размер,
   - тип (photo/video),
   - дата (если доступно из ответа).
3. Реализовать скачивание файла через HTTP:
   - `http://192.168.42.1/DCIM/...`.
4. Сохранять в локальное хранилище приложения и показывать прогресс загрузки.
5. Добавить кнопку `Open`/`Share` для проверки результата на устройстве.

## 8) Минимальные сценарии записи/съемки (для проверки управления)
1. Кнопки:
   - Start/Stop record (`513`/`514`),
   - Shot photo (`16777220` с базовым пресетом).
2. Отображать текущий статус записи:
   - через нотификации,
   - через периодический `Get Record Status` (`16777241`) при активной записи.
3. После завершения записи/фото обновлять список файлов.

## 9) Обработка ошибок и устойчивость (минимально необходимая)
1. Базовая таблица ошибок по `rval`: `-3`, `-4`, `-21`.
2. При `-4` (token invalid): перезапуск сессии автоматически.
3. При обрыве TCP: автоматический reconnect + повтор start session.
4. Если камера занята (`-21`) при чтении настроек: повтор запроса с задержкой.
5. Все ошибки показывать в debug-логе и коротким сообщением в UI.

## 10) Отладка по USB и ежедневный рабочий цикл
1. Подключить Android по USB, включить USB debugging.
2. Запускать через Android Studio/Gradle install.
3. Проверять логи:
   - `adb logcat` по тегам `CameraSocket`, `SessionManager`, `CameraRepository`.
4. Держать «чеклист ручной проверки» (без тестов):
   - подключение,
   - смена 2–3 настроек,
   - старт/стоп записи,
   - скачивание файла.

## 11) Приоритет реализации (чтобы быстро получить value)
1. **День 1–2:** транспорт TCP + session start + init.
2. **День 3:** UI подключения и статус камеры.
3. **День 4:** настройки (получение options + set).
4. **День 5:** листинг файлов + HTTP download.
5. **День 6:** запись/фото кнопки + стабилизация reconnect/ошибок.
6. **День 7:** шлифовка live-плеера, UX, логов и ручной прогон полного сценария.

## 12) Definition of Done для прототипа
Готово, если на Android-устройстве в USB debug режиме разработчик может:
1. Подключиться к камере и поднять сессию.
2. Прочитать и изменить минимум 3 настройки камеры.
3. Увидеть live RTSP-поток на экране приложения и перезапустить его после reconnect.
4. Запустить/остановить запись или сделать фото.
5. Получить список файлов с камеры.
6. Скачать файл с камеры в телефон и открыть его локально.
