# Что ещё осталось сделать

Документ фиксирует текущие пробелы прототипа и конкретные шаги до полноценно проверяемого DoD из `docs/IMPLEMENTATION_PLAN.md`.

## 1) Стабилизировать Android-проект как собираемый артефакт

1. Добавить Gradle Wrapper (`gradlew`, `gradlew.bat`, `gradle/wrapper/*`) и проверить сборку через wrapper.
2. Добавить отсутствующие Android-ресурсы (`strings.xml`, тема, иконки, базовые цвета), чтобы проект гарантированно собирался в Android Studio/CLI.
3. Проверить корректность `minSdk/targetSdk/compileSdk` и зафиксировать рабочие версии AGP/Kotlin/Compose в одном месте.
4. Добавить базовые product flavors/build types (минимум `debug`) и проверить `assembleDebug`.

## 2) Доработать транспорт и протокол камеры до production-like поведения

1. Ввести строгую модель request/response с корреляцией ответов по `msg_id + token + seq` (или аналогичному признаку), чтобы не было race-condition при параллельных событиях.
2. Реализовать корректный парсинг фреймов TCP при любом split/concat пакетов (включая частичный JSON и несколько JSON в одном буфере).
3. Добавить централизованный retry/backoff policy:
   - reconnect TCP;
   - повтор команд для recoverable-ошибок;
   - ограничение по числу попыток + причину отказа в лог.
4. Уточнить heartbeat-сценарий J22: перезапуск таймера после исходящих команд и реакцию на таймаут heartbeat-ответов.
5. Реализовать детальную карту ошибок `rval` (`-3`, `-4`, `-21` и др.) с автодействиями (перезапуск сессии, отложенный retry и т.д.).

## 3) Завершить сценарий сессии и инициализации

1. После `257` валидировать обязательные флаги состояния камеры (`album`, `fwupdate`, `sdformat`, `live`, и т.п.) и показывать их в UI.
2. Реализовать «инициализация завершена» как явное состояние автомата, а не только последовательную отправку команд.
3. Добавить восстановление сессии после `token invalid` и после разрыва сети/сокета.

## 4) Довести live-preview до требований DoD

1. Подключить RTSP-плеер с обработкой состояний `Buffering/Playing/Error` на событиях плеера, а не только на кнопках.
2. Автоматически останавливать live при потере сети/сессии и показывать понятную причину.
3. Добавить автоподнятие live после успешного reconnect + re-init.
4. Добавить отдельный live-лог (старт/стоп, ошибка декодера, время восстановления).

## 5) Реально связать UI настроек с ответами камеры

1. Загружать options по `msg_id=9` из фактических ответов камеры, а не из заглушек.
2. Обновлять текущее значение setting из `msg_id=3`.
3. Для `related > 0` делать повторную подгрузку зависимых settings/options.
4. Проверить минимум 3 изменяемых настройки из DoD на реальном устройстве.

## 6) Файлы: листинг/скачивание/открытие

1. Парсить реальный формат ответа `1282` с пагинацией `from/to`.
2. Для загрузки файла:
   - обработать HTTP-ошибки и отмену;
   - проверять свободное место;
   - сохранять в MediaStore/Downloads (где уместно).
3. Добавить кнопки `Open` и `Share` для каждого загруженного файла в UI.
4. После записи/фото делать авто-refresh списка файлов.

## 7) Запись/фото и актуальный статус

1. Реализовать отображение record status из нотификаций + периодический `16777241` при активной записи.
2. Явно обновлять UI-состояние «идёт запись/простаивает».
3. Добавить блокирующую логику для конфликтующих действий (например, повторный start record во время записи).

## 8) Технический лог и наблюдаемость

1. Привести логи к единому формату и тегам (`CameraSocket`, `SessionManager`, `CameraRepository`, `Live`).
2. Добавить экспорт debug-лога (например, share TXT) для полевой отладки.
3. Подготовить короткий чеклист ручной регрессии (connect/settings/live/record/download).

## 9) Минимальная проверка качества перед каждым пушем

1. Прогонять `./gradlew :app:assembleDebug`.
2. Прогонять `./gradlew :app:lintDebug`.
3. Вести список ручных проверок на телефоне (ADB + реальная камера) и фиксировать результат.

## Целевое подтверждение DoD

Считать задачу завершённой только когда на физическом Android-устройстве воспроизведены все пункты DoD из плана:
1. подключение + сессия;
2. изменение минимум 3 настроек;
3. live RTSP и его восстановление после reconnect;
4. start/stop record или photo;
5. получение списка файлов;
6. скачивание и локальное открытие файла.
